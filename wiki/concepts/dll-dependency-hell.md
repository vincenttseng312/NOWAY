# DLL 依賴衝突（"DLL Hell"）

當一個程式或 DLL 呼叫另一個 DLL 中的函式時，就形成了依賴關係 —— 該程式不再是自我完備的，一旦這個依賴在背後發生變化，程式就可能壞掉。這類問題俗稱「DLL Hell」。

## 什麼情況會破壞依賴關係

當程式所依賴的 DLL 發生以下狀況時，程式可能無法正常執行：
- 被升級成不相容的新版本，
- 被修補／修復，但改變了行為，
- 被舊版覆蓋，或
- 從電腦上被移除。

由於一個共用 DLL 可能同時服務許多程式，某個程式的更新或解除安裝，可能會在不知不覺中破壞另一個剛好共用同一 DLL 的程式 —— 在 Win32 DLL 這個層級上，並沒有內建機制去強制保證所有使用者都能與磁碟上目前的版本相容。

## Windows 的緩解措施（Windows 2000 以後）

- **Windows File Protection**：作業系統會阻擋未經授權地替換或刪除被標記為系統 DLL 的檔案；當安裝程式試圖動到這類檔案時，系統會檢查其數位簽章是否有效。
- **私有 DLL（Private DLL）**：程式可以把自己的 DLL 副本放在自己的程式資料夾中，藉此與共用 DLL 的變動隔離；可透過內嵌在 DLL 中的版本資訊，或一個空白的 `.local` 標記檔來釘選。這會告訴作業系統優先使用本地副本，而非共用／系統版本。
- **固定的搜尋順序**：DLL 解析時一律先檢查應用程式所在資料夾，再檢查 Windows 系統／Windows 資料夾 —— 這正是私有 DLL 與並存副本得以運作的關鍵，詳見 [[dynamic-link-library]] 中的搜尋順序章節。

這些都是緩解措施，而非結構性的根本解決方案 —— 它們降低了出問題的機率，但仍仰賴人為紀律（回溯相容性、正確使用 `.local`），而非由平台本身強制保證。

## 結構性的根本解法：.NET assembly

依微軟自身的論述框架（見 [[dll-ms-learn]]），這些緩解措施後來並非被「補強」，而是被 .NET assembly 模型「取代」—— 版本控管與隔離機制直接內建於檔案格式本身，而不是靠作業系統層級的保護機制事後補上。比較細節見 [[dotnet-assembly]]。
